npet2 (npet_extraction)

npet2 is a standalone ribosome nascent peptide exit tunnel (NPET) geometry pipeline. Given an RCSB structure ID (or a local mmCIF) it:

  • acquires landmarks (PTC + constriction site)
  • defines a canonical tunnel axis frame (PTC → constriction = +Z, “C0” frame)
  • computes empty space inside an exterior ribosome shell within a cylinder around the tunnel axis
  • clusters the empty-space voxels to isolate the NPET void
  • optionally runs a refinement pass on a smaller ROI at higher resolution
  • meshes the tunnel void (marching cubes + smoothing) and validates watertightness
  • reports tunnel lining chains by proximity of atoms to the final mesh

Defaults are “API-first”: profile + landmarks come from ribosome.xyz, and mmCIF is downloaded from RCSB unless you provide a local mmCIF.

Installation (local)

Requirements

  • Python >= 3.10 (Docker image uses 3.11)
  • On Linux you typically also need system libs for PyVista/VTK rendering deps:
    • libgomp1 and libglib2.0-0 (these are installed in the Dockerfile; for local installs your distro packages may vary)

Install

git clone <this-repo>
cd <this-repo>

python -m venv .venv
source .venv/bin/activate

pip install -U pip setuptools
pip install .

Verify

npet2 --help
npet2 show-config

Installation & usage (Docker)

Build

docker build -t npet2:latest .

Run with a mounted data directory

All outputs, caches, and downloaded mmCIFs are written under /data inside the container.

mkdir -p ./npet2_data
docker run --rm -it \
  -v "$(pwd)/npet2_data:/data" \
  npet2:latest run 5NWY

Use a custom ribosome.xyz-compatible API endpoint

docker run --rm -it \
  -e NPET2_RIBOXYZ_API_URL="https://api.ribosome.xyz" \
  -v "$(pwd)/npet2_data:/data" \
  npet2:latest run 5NWY

docker compose

docker-compose.yml mounts ${NPET2_DATA:-./npet2_data} to /data and lets you override the API URL.

# default volume ./npet2_data -> /data
docker compose run --rm npet2 run 5NWY

# override data dir + API url
NPET2_DATA=/absolute/path/to/npet2_data \
NPET2_RIBOXYZ_API_URL=https://api.ribosome.xyz \
docker compose run --rm npet2 run 5NWY

Environment variables

These are the most relevant knobs for paths and external dependencies (see libnpet/core/config.py):

Variable Default Meaning
NPET2_ROOT ~/.npet2 (local) / /data (docker) Root for all npet2 data
NPET2_RUNS_ROOT $NPET2_ROOT/runs Where run outputs are written (this is effectively your “output log root”)
NPET2_CACHE_ROOT $NPET2_ROOT/cache Stage cache directory (re-uses expensive outputs like exterior shell)
NPET2_POISSON_RECON_BIN $NPET2_ROOT/bin/PoissonRecon Optional external binary location (used by legacy/optional reconstruction helpers)
NPET2_RIBOXYZ_API_URL http://localhost:8000 (local) / https://api.ribosome.xyz (docker) Base URL for profile/landmark fetches

Where does output get logged? The pipeline prints to stdout/stderr. Persisted run state and artifacts are written under:

  • ${NPET2_RUNS_ROOT}/{RCSB_ID}/{RUN_ID}/...

The canonical “run record” is:

  • ${NPET2_RUNS_ROOT}/{RCSB_ID}/{RUN_ID}/manifest.json

Output structure

A single run lives at:

{NPET2_RUNS_ROOT}/{RCSB_ID}/{RUN_ID}/
├── manifest.json
├── stage/
│   ├── 00_inputs/
│   ├── 10_landmarks/
│   ├── 20_exterior_shell/
│   ├── 30_region_atoms/
│   ├── 40_empty_space/
│   ├── 50_clustering/
│   ├── 55_grid_refine/
│   ├── 70_mesh_validate/
│   └── 80_tunnel_lining/
├── tunnel_mesh.ply
├── tunnel_mesh_ascii.ply
├── tunnel_lining.json
└── tunnel_lining.cif

manifest.json

A machine-readable run ledger (libnpet/core/manifest.py) containing:

  • inputs fingerprints (what files/API endpoints were used)
  • resolved config parameters
  • per-stage status/timing + parameters
  • a list of produced artifacts (type, stage, path, metadata)

Stage folders (what to expect)

  • stage/00_inputs/

    • atom_xyz.npy: all atoms loaded from the mmCIF (float32, shape Nx3)

  • stage/10_landmarks/

    • ptc.json: {"location":[x,y,z]} (Å)
    • constriction_site.json: {"location":[x,y,z]} (Å) plus metadata in manifest

  • stage/20_exterior_shell/

    • ribosome_ptcloud.npy: atom point cloud used for the shell
    • alpha_shell.ply: watertight exterior shell mesh (best-effort)
    • alpha_shell_quality.json: manifold/open-edges/bounds diagnostics

  • stage/30_region_atoms/

    • region_atom_xyz.npy: seed atoms (walls-only chains) within the cylinder
    • region_atom_xyz_occ.npy: occupancy atoms used for empty-space exclusion
    • atom_selection_policy.json: included/excluded chains and reasons

  • stage/40_empty_space/

    • empty_points_level_*.npy: empty voxel centers inside shell & cylinder
    • empty_points_level_*_preclip.npy: before shell “inside” filtering
    • For edt backend levels: occupancy_grid_*_{data.npy,spec.json}, empty_mask_*_{data.npy,spec.json}
    • clipping_note.json: whether shell was watertight and how clipping was done

  • stage/50_clustering/

    • coarse/ and refine/ DBSCAN dumps:

      • points.npy, labels.npy, index.json, and cluster_idX.npy

    • largest_cluster.npy: selected cluster from coarse pass (by proximity to constriction)

    • refined_cluster.npy: selected cluster from refine pass

    • mesh_level_0.ply (+ *_ascii.ply, *_pre_smooth.ply) if meshing enabled

  • stage/55_grid_refine/

    • ROI + refined DBSCAN/void masks at ~0.5Å (default):

      • roi_bbox_c0.json, grid_spec_level_1.json
      • occupied_mask_level_1.npy, void_mask_level_1.npy
      • selected_void_component_mask_level_1.npy (preferred for meshing)
      • coarse/ and refine/ DBSCAN outputs
      • refined_surface_points_level_1.npy
      • mesh_level_1.ply (+ ascii / pre_smooth) if enabled

  • stage/70_mesh_validate/

    • npet2_tunnel_mesh.ply (final chosen mesh; must be watertight)
    • comparison_mesh_level_{0,1}*.ply copies for debugging

  • stage/80_tunnel_lining/

    • tunnel_lining.json: chains within a distance threshold of the mesh
    • tunnel_lining.cif: subset mmCIF containing qualifying chains

Convenience copies in run root

After validation/lining:

  • tunnel_mesh.ply, tunnel_mesh_ascii.ply
  • tunnel_lining.json, tunnel_lining.cif
  • (and sometimes the level meshes copied up as well)

Dependency inputs: local files vs ribosome.xyz API

npet2 supports three dependency types:

  1. mmCIF structure (atoms)
  2. ribosome profile (chain metadata & assembly map)
  3. landmarks: PTC + constriction site

Default behavior (no files provided)

  • mmCIF: downloaded from RCSB and cached under ${NPET2_ROOT}/mmcif/{RCSB_ID}/{RCSB_ID}.cif
  • profile: fetched from ribosome.xyz-compatible API
  • landmarks: fetched from ribosome.xyz-compatible API

Mixed behavior

You can provide any subset as local files. Any omitted dependency falls back to API (except mmCIF, which falls back to RCSB download).

Examples

API-only (recommended default)

npet2 run 5NWY --api-url https://api.ribosome.xyz

Local mmCIF, API for profile + landmarks

npet2 run 5NWY --mmcif /path/to/5NWY.cif

Local profile + landmarks, mmCIF auto-download

npet2 run 5NWY \
  --profile /path/to/5NWY_profile.json \
  --ptc /path/to/5NWY_PTC.json \
  --constriction /path/to/5NWY_CONSTRICTION_SITE.json

All-local

npet2 run 5NWY \
  --mmcif /path/to/5NWY.cif \
  --profile /path/to/5NWY_profile.json \
  --ptc /path/to/5NWY_PTC.json \
  --constriction /path/to/5NWY_CONSTRICTION_SITE.json

Required JSON schemas

These schemas are enforced via pydantic models in libnpet/core/ribosome_types.py and adapters in libnpet/adapters/standalone_providers.py.

1) PTC JSON (--ptc)

Must validate as PTCInfo:

{
  "location": [x, y, z]
}
  • exactly 3 numeric values (Å)
  • extra fields are allowed and ignored

2) Constriction JSON (--constriction)

Must validate as ConstrictionInfo:

{
  "location": [x, y, z]
}
  • exactly 3 numeric values (Å)
  • extra fields are allowed and ignored

3) Profile JSON (--profile)

Must validate as RibosomeProfile:

  • top-level:

    • rcsb_id (string)
    • mitochondrial (bool, optional)
    • proteins, rnas, other_polymers (lists, optional)
    • assembly_map (optional but required if you use occupancy_chain_mode="assembly_all")

Minimal acceptable example:

{
  "rcsb_id": "5NWY",
  "mitochondrial": false,
  "proteins": [
    {
      "auth_asym_id": "A",
      "assembly_id": 0,
      "nomenclature": [],
      "rcsb_pdbx_description": "some protein",
      "entity_poly_seq_length": 120
    }
  ],
  "rnas": [
    {
      "auth_asym_id": "r",
      "assembly_id": 0,
      "nomenclature": [],
      "rcsb_pdbx_description": "23S rRNA",
      "entity_poly_seq_length": 2900
    }
  ],
  "other_polymers": [],
  "assembly_map": [
    {
      "rcsb_id": "5NWY",
      "polymer_entity_instances": [
        {"entity_id": "1", "auth_asym_id": "A"},
        {"entity_id": "2", "auth_asym_id": "r"}
      ],
      "nonpolymer_entity_instances": null
    }
  ]
}

Notes:

  • nomenclature can be strings; they are coerced into internal enums.
  • If you fetch profile from API, the adapter flattens the assembly_map nesting to match this schema.

CLI reference

The npet2 CLI is defined in libnpet/__main__.py and exposes three commands:

npet2 setup

Builds/install external binaries (currently PoissonRecon) under $NPET2_ROOT/bin/PoissonRecon.

Options

  • --data-dir DIR: overrides $NPET2_ROOT for where build artifacts go

npet2 show-config

Prints default RunConfig as JSON (all pipeline numerical parameters).

npet2 run

Runs the full pipeline on one or more structures.

Target selection

  • npet2 run 5NWY 7K00 ... : positional IDs
  • --rcsb_id ID : single ID alternative
  • --from-file FILE : file containing one ID per line

Data sources (dependency inputs)

  • --api-url URL : ribosome.xyz API base URL (overrides NPET2_RIBOXYZ_API_URL)
  • --mmcif PATH : local mmCIF path (otherwise RCSB download + cache)
  • --profile PATH : local profile JSON (otherwise API)
  • --ptc PATH : local PTC JSON (otherwise API)
  • --constriction PATH : local constriction JSON (otherwise API)
  • --data-dir DIR : set NPET2 root for mmcif cache/build/runs defaults

Output control

  • --output-dir DIR : override where runs are written (maps to NPET2_RUNS_ROOT)

Parallelism

  • --workers, -j N : run multiple RCSB IDs in parallel (each is an independent run)

Advanced config overrides (stage-by-stage meaning)

These flags override fields in RunConfig (libnpet/core/config.py) and affect specific stages:

Region definition (Stage10/Stage30/Stage40)

  • --cylinder-radius A → cylinder_radius_A Radius of the cylinder around the tunnel axis used to select atoms and define the search volume.
  • --cylinder-height A → cylinder_height_A Cylinder extent along +Z (C0 frame) from the constriction direction.
  • --ptc-extension A → cylinder_ptc_extension_A Extends the cylinder into negative Z (below PTC) to include PTC-adjacent space.

Grid resolution (Stage40 + Stage50)

  • --voxel-size A Sets level_0 grid voxel size (creates a single grid_levels=[level_0] entry). This impacts how empty space voxels are generated and thus DBSCAN input density.
  • --refine-voxel-size A → refine_voxel_size_A Voxel size for Stage55 refinement ROI (default 0.5Å).

Clustering (Stage50, Stage55)

  • --dbscan-coarse-eps A → dbscan_level0_coarse_eps_A Coarse DBSCAN radius (Å) for initial cluster merge.
  • --dbscan-coarse-min-samples N → dbscan_level0_coarse_min_samples Density threshold for coarse DBSCAN.
  • --dbscan-refine-eps A → dbscan_level0_refine_eps_A Refine DBSCAN radius (Å) applied to the selected coarse cluster.
  • --dbscan-refine-min-samples N → dbscan_level0_refine_min_samples Density threshold for refine DBSCAN.

Meshing (Stage50/Stage55/Stage70)

  • --no-mesh Disables mesh generation at level_0 and level_1 (mesh_level0_enable=False, mesh_level1_enable=False). Stage70 will then fail because it requires a mesh; use this mainly for debugging earlier stages.

Refinement (Stage55)

  • --no-refine Intended to skip Stage55 refinement. (Note: if Stage55 is always in the pipeline list, this flag must be honored by code inside Stage55 or by constructing a reduced pipeline. If your current branch still runs Stage55 unconditionally, this flag may not have effect.)

Tunnel lining (Stage80)

  • --lining-proximity A → lining_proximity_A Atom-to-mesh distance threshold (Å) for marking chains as “lining”.
  • --no-nonpolymers → lining_include_nonpolymers=False Excludes nonpolymer ligands/ions from lining output.
  • --include-waters → lining_include_waters=True Includes water molecules in lining output.
